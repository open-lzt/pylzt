# Журнал живой верификации кодогена

`tests/pylzt/e2e/test_live_read.py` (маркер `e2e`, требует `LZT_E2E_TOKEN`,
исключён из прогона по умолчанию) автоматически находит каждый GET-эндпоинт без
аргументов и прогоняет его против реального API. Каждый сгенерированный файл
несёт докстринг `Generated by forge — DO NOT EDIT` (генератор,
`dev/codegen/generator.py`, называется **forge**) — этот документ отслеживает,
какие эндпоинты уже верифицированы вживую и что было реально не так, когда они
не проходили.

Запуск: `LZT_E2E_TOKEN=<token> python -m pytest -m e2e tests/pylzt/e2e/test_live_read.py -v`

## Исправлено и верифицировано вживую (2026-07-05)

- **Критично: каждый сгенерированный метод `forum_*` тихо роутился на хост
  MARKET.** `render_method_skeleton` никогда не эмитил `__api__`/`__rate_class__`,
  поэтому все ~90 сгенерированных forum-операций по умолчанию получали
  `ApiTarget.MARKET`/`RateClass.GENERAL` — либо честно падали в 404, либо, что
  хуже, попадали на не связанный с ними market-эндпоинт с тем же путём и
  парсили мусор в forum-модель (именно из-за этого `ChatboxIndex` выглядел как
  баг "неверный тип поля" — на деле это был `/chatbox` на неверном хосте).
  Исправлено в `dev/codegen/generator.py` (`render_method_skeleton` теперь
  принимает `api` и эмитит оба class-var для `api="forum"`). Регенерируйте,
  чтобы подтвердить: `python -m dev.codegen build --api market --api forum`.
- `pylzt/models/forum/chatbox_index_response.py` (`ban`), `.../room.py`
  (`deal_id`) — на практике nullable, были объявлены обязательным `str`.
- `pylzt/models/forum/forum.py` — `forum_rules_thread_id` nullable;
  `forum_moderators` — список полных объектов пользователя, а не юзернеймов
  (ослаблено до `list[dict[str, Any]]`).
- `pylzt/models/forum/forums_list_tab.py` — `title`/`isHidden` иногда
  отсутствуют.
- `pylzt/models/market/cart_get_response.py` (используется
  `CartGet`/`ListFavorites`/`ListOrders`/`CategoryAll`) — `items` — это список
  полных объектов, а не строк (ослаблено до `list[dict[str, Any]]`, не
  смоделировано точно — см. ниже); `totalItemsPrice` — это
  `str | int | None` в зависимости от запроса, а не обязательный `str`.
- `pylzt/models/market/category_response.py` (общая база, которую возвращает
  КАЖДЫЙ эндпоинт category-search) — та же nullability для `totalItemsPrice`,
  исправлено один раз для всего семейства.
- **`canResellItemAfterPurchase` + `bumpSettings`, все 19 моделей category
  item** (`discord_item.py`, `ea_item.py`, ... `steam_item.py`) — оба поля
  объявлены обязательными, оба на практике отсутствуют в заметной доле живых
  листингов по каждой категории. Патч применён батчем (не поднято в общую
  базу — хостинг по Rule-2 кодогена поднимает только *ведущую* последовательность
  идентичных полей, а эти два стоят в середине списка, поэтому каждая
  категория объявляет их заново; ослабление каждого по отдельности — верное
  решение, не костыль). Снизило общий объём ошибок валидации по категориям
  с ~33 упавших до 30 упавших, но большинство категорий всё ещё падают на
  своих оставшихся уникальных полях — см. ниже.
- `pylzt/models/lot.py` — реальный баг, не связанный со спекой OpenAPI:
  `OrderBy` импортировался только под `TYPE_CHECKING`, поэтому pydantic не мог
  разрешить forward-ref `LotFilter.order_by` в рантайме, и
  `Client.list_lots(...)` бросал `PydanticUserError` на каждом вызове. Не был
  пойман раньше, потому что ни один тест не прогоняет путь построения схемы с
  реальным event loop'ом. Теперь это реальный top-level импорт.

Все шесть файлов моделей выше лишились авто-заголовка (теперь там написано
"Hand-patched" и объяснение почему) — это намеренно: `_guard_no_clobber` в
`install()` теперь ПРОПУСТИТ эти файлы на следующем
`dev.codegen build --api {market,forum}`, который их затронет (с предупреждением,
исключены из установки — остальная часть батча всё равно установится) вместо
тихой регенерации неверного типа. Решается перепроверкой по живому захвату и
либо повторным продвижением, либо обновлением этого файла — никогда
принудительной перезаписью.

## Аудит полей по категориям — завершён (2026-07-05, продолжение сессии)

Бэклог из 30 эндпоинтов, оставленный проходом выше, теперь полностью закрыт:
**65 passed, 1 skipped, 5 xfailed, 0 failed** на полном наборе `e2e`.
Забюджетировано как отдельная задача согласно заметке выше; сделано в три
волны на живых данных (тип никогда не угадывался без предварительного
захвата `exc.errors(include_input=True)`):

- **Общие сквозные поля, все 21 модель category-item**: `isIgnored`,
  `hasPendingAutoBuy`, `note_text`, `email_provider`,
  `sold_items_category_count`, `restore_items_category_count`,
  `feedback_data` → nullable; `priceWithSellerFee` `int → float`
  (дробные живые цены); `guarantee` унифицирован на общей модели
  `ItemGuarantee` (был просто `str` в 19 из 21 файла) и сам сделан nullable;
  `CategoryDiscordItemSeller.restore_percents` `str → int` (баг
  copy-paste — у соседнего `ItemSeller.restore_percents` всегда был `int`),
  также сделан nullable на обеих моделях seller.
- **Модели значений динамических карт легитимно теряют поля**: у моделей
  значений `eg_games` (EpicGames), `uplay_games` (Uplay), `ea_games` (EA) были
  обязательные поля (`category_id`/`ru`/`hits_count`,
  `pvpTimePlayed`/`pveTimePlayed` и т. д.), которые отдельные игры опускают —
  ослаблено до nullable. То же для модели статистики по сезону
  `fortnitePastSeasons` (Fortnite) — обычный список, не карта, но та же форма
  "некоторые поля отсутствуют в отдельных записях". Некоторые из этих
  карт/списков ТАКЖЕ приходят как пустой `list`/`[]` вместо ожидаемого
  `dict`/модели на аккаунтах без контента (`eg_games`, `uplay_games`,
  `battlenetGames`, `valorantInventory`, `lolInventory`) — расширено до
  `T | list[Any] | None`, а не просто до nullable.
- **Однозначно неверные типы, не просто nullability**: `discord_nitro_type` —
  строковый литерал вроде `"none"`, не int (`str | None`, ещё не `StrEnum` —
  наблюдалось только одно значение вживую, пересмотреть, если появятся
  другие); элементы `r6Skins` / `eg_code_redemption_history` — полные объекты
  (`item_id`/`name`/... / `description`/`redeemedDate`/...), не простые
  строки (`list[dict[str, Any]]`, тот же прецедент "не смоделировано точно",
  что и `cart_get_response.py`); `socialclub_games` — JSON-закодированная
  строка со слагами, не список богатых объектов (`list[str]` + `before`-валидатор,
  который делает `json.loads`); `steam_bans` — `dict[str, str]`
  (appid → причина), но `""`, когда пусто (валидатор мапит `"" → None`);
  `tarkov_kd` — строка-десятичная дробь (`float`, достаточно простого
  приведения типа); `System.providers` — динамическая карта, ключи которой —
  реальные коды провайдеров (`SOL_USDC`, `BSC_BNB`, ...), а не
  фиксированный набор `{BEP20,TRC20,...}`, заявленный в спеке;
  реальный ключ на проводе у `stats.noMarket` — строчный `nomarket` (тихо
  всегда был `None` под старым алиасом, а не просто "отсутствует" — исправлено
  через `AliasPath`, не только через nullability); хак с буквальным ключом
  "domain.zone" в `imap_data` у `ProfileGet` никогда ничему реальному не
  соответствовал — заменён на настоящую динамическую карту
  `dict[str, UserDomainZone]`, ключи которой — реальные IMAP-домены.
- **`ForumsGetFeedOptions`** получил собственную выделенную модель
  `ForumFeedOption` (новый файл) вместо переиспользования/ослабления общей
  модели `Forum` — подтверждено вживую, что `/forums/feed-options` стабильно
  возвращает более лёгкую форму (без
  `forum_thread_count`/`forum_post_count`/`forum_prefixes`/`thread_default_prefix_id`/
  `thread_prefix_is_required`, плюс одно поле, которого нет у `Forum`:
  `has_children`) по всем 353 сэмплированным записям, а сама `Forum` всё ещё
  корректно требует много полей для `ForumsList`, так что ослабление её на
  месте тихо сломало бы реальные гарантии этого эндпоинта.
- Всё остальное из исходного списка "известно сломанных"
  (`ForumsGrouped`, `FormsList`, `ChatboxGetLeaderboard`,
  `PaymentsPayoutServices`, `ProfileClaims`, `ProfileGet`, `ListStates`,
  `ListViewed`) — прямолинейные исправления отсутствующих/неверно
  типизированных полей, тот же паттерн, что и выше;
  `ProfileGet.convertedBalance` — `float`, не `int` (`int_from_float`,
  тот же класс, что и `priceWithSellerFee`).

Каждый затронутый файл модели несёт датированную заметку hand-patch "N-й
проход", объясняющую, что показали живые данные и почему — прочитайте
докстринг самого файла, прежде чем трогать поле, которое он покрывает. Живые
листинги дрейфуют между прогонами (разная выборка активных лотов при каждом
запуске набора) — ожидайте, что даже после этого прохода на свежем прогоне
время от времени будет всплывать новое разовое поле; паттерн исправления
всегда один и тот же (захватить реальное значение через
`exc.errors(include_input=True)`, не угадывать тип).

## Добавлена универсальная поддержка `/batch` (2026-07-06)

`market_batch_requests.Batch` и `forum_batch_requests.BatchExecute` были
сгенерированы кодогеном с **нулём полей тела запроса** — тело POST /batch по
спеке — это сырой массив объектов job, форма, которую генератор методов с
плоскими полями захватить не может, поэтому оба стаба были тихо
неработоспособны (вызов отправлял пустой массив). Их угаданная модель ответа
(`Jobs`/`JobsJobId`) тоже была неверна — тот же класс бага, что и переиспользование
`Forum` для `ForumsGetFeedOptions`: `jobs` моделировался как одно
фиксированное поле `job_id`, тогда как реальная форма на проводе — это
динамическая карта, ключи которой — id job'ов.

Добавлены `BatchJob`/`BatchJobResult` +
`build_generic_batch_request`/`parse_generic_batch_body` в `lib/batch.py`
(уже существовавшая пара `_build_batch_request`/`_parse_batch_body`,
специфичная для лотов, теперь тонкие обёртки над ними — одна реализация
механики на проводе). `Batch`/`BatchExecute` пропатчены вручную, чтобы
принимать реальное поле `jobs: Sequence[BatchJob]` (со значением по умолчанию
`()` только для того, чтобы всё ещё сгенерированные, всё ещё не подключённые
стабы фасада `GeneratedMarketFacade.batch()` / `GeneratedForumFacade.batch_execute()`
оставались валидными по типам — реальные вызовы всегда передают `jobs`
явно; этим двум точкам вызова фасада понадобился документированный
`# type: ignore[arg-type]`, так как их заявленный тип возврата `-> Jobs`
больше не совпадает, но сами файлы фасада в остальном оставлены
нетронутыми — ручной патч целого сгенерированного файла на ~1500 строк ради
двух строк заблокировал бы обновление кодогеном остальных ~100 методов в нём).
Добавлен `Client.execute_batch(methods: Sequence[BaseMethod[T]]) -> list[T]`:
**разнородные** методы батчатся вместе без проблем (подтверждено вживую:
`ListCategories` + два `GetLot` в одном POST /batch, каждый распарсен своим
`parse_response`) — более ранний вопрос "это 1 тип или много" теперь отвечен
вживую, а не угадан.

Ещё две находки, обнаруженные только вживую при подключении этого, обе
незаметно давали бы неверный результат без живой проверки:

- **Id job'а `"0"` на сервере трактуется как пустой** (почти наверняка
  особенность PHP/XenForo `empty("0") == true`) — ответ откатывается к тому,
  чтобы ключевать результат этого job'а по его URI вместо отправленного id.
  `execute_batch` использует id job'ов с индексацией от 1 (`"1"`, `"2"`, ...),
  никогда `"0"`, чтобы обойти это. Уже существовавший путь батча по лотам
  никогда с этим не сталкивался (реальные id лотов никогда буквально не
  равны `0`).
- **Запись хита батча НЕ является `{"_job_result": "ok", "item": {...}}`
  единообразно** — это `{"_job_result": "ok", **<собственное top-level тело
  ответа батчуемого эндпоинта>}`. Job'ы `GetLot` выглядят обёрнутыми в
  `"item"` только потому, что у собственного ответа `/{item_id}` на одиночный
  вызов легитимно есть `"item"` как top-level ключ; батчуемый job
  `ListCategories` возвращается как `{"_job_result": "ok", "categories": [...]}`
  без какого-либо ключа `"item"` вообще. `BatchJobResult.item` теперь снимает
  только маркеры `_job_result`/`_job_error` и передаёт остальное
  собственному `parse_response` метода без изменений — именно это делает
  разнородный батчинг рабочим без спецкейсов на каждый эндпоинт.

## Отдельный, ранее существовавший баг найден (не часть этого прохода) — исправлен 2026-07-07

`tests/pylzt/test_methods_declarative.py` (6 тестов) падали против текущего
`BaseMethod` — они утверждали конструирование в стиле dataclass
(`GetLot(ItemId(7))` позиционно, `dataclasses.is_dataclass(method)`), что не
было верным с момента коммита `7016a77` ("convert the method layer from
frozen dataclass to Pydantic"). Гейт кодогена (`pipeline.validate()`)
запускает только `ruff` + `mypy` + smoke-тест импорта — **он никогда не
запускает `pytest`**, поэтому эта регрессия ушла в прод тихо.

**Исправлено 2026-07-07**: тесты теперь конструируют через keyword-аргументы
и проверяют `pydantic.ValidationError` для frozen-проверки, соответствуя
контракту после `7016a77`. `.github/workflows/ci.yml` теперь запускает
`ruff check` + `ruff format --check` + `mypy` + `pytest` на каждом push/PR,
поэтому такой класс поломок теперь падает в CI, а не уходит в прод тихо.
Сам гейт кодогена не изменён (по-прежнему только ruff+mypy+import) — CI
служит подстраховкой для набора тестов, а не гейтом.

## Не-JSON ответ 200 тихо терялся, исправлено 2026-07-09

Три эндпоинта объявляют свой ответ 200 как `text/html`/`text/plain` со схемой
`{"type": "string"}`, не `application/json`: `ListDownload`
(`/user/{type}/download`), `ManagingSteamPreview` (`/{item_id}/steam-preview`),
`PublicCountLinesPlain` (`/countLinesPlain`). Наложились два независимых
бага:

- `_resolve_response_schema` в `dev/codegen/generator.py` смотрел только на
  `content["application/json"]`, поэтому все три резолвились без схемы вообще
  и генерировались как `BaseMethod[Any]` — технически не ошибка, но
  скрывающая реальный баг.
- `_decode` в `transport/session.py` всегда вызывал `response.json()` и
  возвращал `{}` при сбое без фолбэка — так что даже до исправления типизации
  вызов любого из этих трёх эндпоинтов тихо отбрасывал реальное тело и
  возвращал пустой словарь.

**Исправлено 2026-07-09**: `_resolve_response_schema` теперь откатывается на
любой заявленный content type, когда `application/json` отсутствует, поэтому
эти три теперь генерируются как `BaseMethod[str]` (проверено корректно —
реген с `_is_object` false на голой схеме `{"type": "string"}` идёт через уже
существующий путь скаляра/`_schema_type`, новая ветка кодогена не нужна).
`Response` получил поле `text: str | None`, устанавливаемое `_decode` всякий
раз, когда тело на проводе — не JSON-объект; ветка `passthrough` в
`BaseMethod.parse_response` предпочитает `text` над (теперь бессмысленным)
`{}` в `body`, когда оно установлено. Это исправление общего механизма, а не
ручной патч под конкретный эндпоинт — любой эндпоинт, который будущее
обновление спеки добавит с той же не-JSON-строковой формой, получит его
бесплатно.

## Справочник по механизму

- **Баг на уровне метода** (неверный `__url__`, неверный `__api__`, неверный
  `__rate_class__`) → чините `dev/codegen/generator.py` (это реальный баг
  генератора, не дрейф спеки) и пересобирайте.
- **Баг на уровне поля модели** (спека заявляет тип, который реальный API не
  соблюдает) → патчите вручную установленный файл `src/pylzt/models/**.py`,
  снимайте его авто-заголовок и фиксируйте исправление здесь. Следующий
  `install()` для этого API пропустит (`_guard_no_clobber`) любой staged-файл,
  конфликтующий с уже написанным вручную — с предупреждением и исключением,
  остальная часть батча всё равно установится — это намеренная защита, а не
  баг; решается сверкой этого документа с моделью, а не принудительной
  перезаписью.
