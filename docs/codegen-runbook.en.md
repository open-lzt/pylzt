# Codegen live-verification runbook

`tests/pylzt/e2e/test_live_read.py` (marker `e2e`, gated on `LZT_E2E_TOKEN`, excluded
from the default run) auto-discovers every zero-argument GET endpoint and runs it against
the real API. Every generated file carries a `Generated by forge — DO NOT EDIT` docstring
(the generator, `dev/codegen/generator.py`, is named **forge**) — this doc tracks which
endpoints have been live-verified, and what was actually wrong when they weren't.

Run it: `LZT_E2E_TOKEN=<token> python -m pytest -m e2e tests/pylzt/e2e/test_live_read.py -v`

## Fixed and verified live (2026-07-05)

- **Critical: every generated `forum_*` method silently routed to the MARKET host.**
  `render_method_skeleton` never emitted `__api__`/`__rate_class__`, so all ~90 generated
  forum operations defaulted to `ApiTarget.MARKET`/`RateClass.GENERAL` — either 404ing
  outright or, worse, hitting an unrelated same-path market endpoint and parsing garbage
  into a forum model (this is what made `ChatboxIndex` look like a "wrong field type" bug
  — it was really `/chatbox` on the wrong host). Fixed in `dev/codegen/generator.py`
  (`render_method_skeleton` now takes `api` and emits both class-vars for `api="forum"`).
  Regenerate to confirm: `python -m dev.codegen build --api market --api forum`.
- `pylzt/models/forum/chatbox_index_response.py` (`ban`), `.../room.py` (`deal_id`) —
  nullable in practice, were declared required `str`.
- `pylzt/models/forum/forum.py` — `forum_rules_thread_id` nullable; `forum_moderators`
  is a list of full user objects, not usernames (loosened to `list[dict[str, Any]]`).
- `pylzt/models/forum/forums_list_tab.py` — `title`/`isHidden` are sometimes absent.
- `pylzt/models/market/cart_get_response.py` (shared by `CartGet`/`ListFavorites`/
  `ListOrders`/`CategoryAll`) — `items` is a list of full objects, not strings (loosened
  to `list[dict[str, Any]]`, not modeled exactly — see below); `totalItemsPrice` is
  `str | int | None` depending on the request, not a required `str`.
- `pylzt/models/market/category_response.py` (the generic base EVERY category-search
  endpoint returns) — same `totalItemsPrice` nullability, fixed once for the whole family.
- **`canResellItemAfterPurchase` + `bumpSettings`, all 19 category item models**
  (`discord_item.py`, `ea_item.py`, ... `steam_item.py`) — both declared required, both
  absent-in-practice on a meaningful fraction of live listings across every category.
  Batch-patched (not hoisted onto a shared base — codegen's Rule-2 hoisting only lifts a
  *leading* run of identical fields, and these two sit mid-list, so each category
  redeclares them individually; loosening each is the correct fix, not a workaround).
  Cut total validation-error volume across categories from ~33 failing to 30 failing, but
  most categories still fail on their own remaining unique fields — see below.
- `pylzt/models/lot.py` — real bug, unrelated to the OpenAPI spec: `OrderBy` was imported
  only under `TYPE_CHECKING`, so pydantic couldn't resolve `LotFilter.order_by`'s forward
  ref at runtime and `Client.list_lots(...)` raised `PydanticUserError` on every call. Not
  caught earlier because no test exercises the schema-building path with a real event
  loop. Now a real top-level import.

All six model files above had their auto-gen header stripped (now say "Hand-patched" and
explain why) — this is deliberate: `install()`'s `_guard_no_clobber` will now SKIP these
files on the next `dev.codegen build --api {market,forum}` that touches them (warned,
excluded from the install — the rest of the batch still installs) instead of silently
regenerating the wrong type. Resolve by re-verifying against a live capture and either
re-promoting or updating this file, never by force-overwriting.

## Per-category field audit — complete (2026-07-05, follow-up session)

The 30-endpoint backlog left by the pass above is now fully closed: **65 passed, 1
skipped, 5 xfailed, 0 failed** on the full `e2e` suite. Budgeted as its own task per the
note above; done in three waves against live data (never guessed a type without an
`exc.errors(include_input=True)` capture first):

- **Shared cross-cutting fields, all 21 category-item models**: `isIgnored`,
  `hasPendingAutoBuy`, `note_text`, `email_provider`, `sold_items_category_count`,
  `restore_items_category_count`, `feedback_data` → nullable; `priceWithSellerFee`
  `int → float` (fractional live prices); `guarantee` unified onto the shared
  `ItemGuarantee` model (was a plain `str` in 19/21 files) and made nullable itself;
  `CategoryDiscordItemSeller.restore_percents` `str → int` (copy-paste bug — sibling
  `ItemSeller.restore_percents` was always `int`), also made nullable on both seller
  models.
- **Dynamic-key map value models legitimately drop fields**: `eg_games` (EpicGames),
  `uplay_games` (Uplay), `ea_games` (EA) value models all had required fields
  (`category_id`/`ru`/`hits_count`, `pvpTimePlayed`/`pveTimePlayed`, etc.) that individual
  games omit — loosened to nullable. Same for `fortnitePastSeasons`' per-season stat
  model (Fortnite) — a plain list, not a map, but the same "some fields absent per
  entry" shape. Several of these maps/lists ALSO arrive as an empty `list`/`[]` instead
  of the expected `dict`/model on zero-content accounts (`eg_games`, `uplay_games`,
  `battlenetGames`, `valorantInventory`, `lolInventory`) — widened to
  `T | list[Any] | None` rather than just nullable.
- **Outright wrong types, not just nullability**: `discord_nitro_type` is a literal
  string like `"none"`, not an int (`str | None`, not yet a `StrEnum` — only one value
  observed live, revisit if more show up); `r6Skins` / `eg_code_redemption_history`
  elements are full objects (`item_id`/`name`/... / `description`/`redeemedDate`/...),
  not plain strings (`list[dict[str, Any]]`, same "not modeled exactly" precedent as
  `cart_get_response.py`); `socialclub_games` is a JSON-encoded string of slugs, not a
  list of rich objects (`list[str]` + a `before` validator that `json.loads`s it);
  `steam_bans` is `dict[str, str]` (appid → reason) but `""` when empty (validator maps
  `"" → None`); `tarkov_kd` is a decimal string (`float`, plain coercion suffices);
  `System.providers` is a dynamic map keyed by real provider codes (`SOL_USDC`,
  `BSC_BNB`, ...), not the fixed `{BEP20,TRC20,...}` set the spec declared;
  `stats.noMarket`'s real wire key is lowercase `nomarket` (was silently always `None`
  under the old alias, not just "missing" — fixed via `AliasPath`, not just nullability);
  `ProfileGet`'s `imap_data` "domain.zone" literal-key hack never matched anything real —
  replaced with a proper `dict[str, UserDomainZone]` dynamic map keyed by actual IMAP
  domains.
- **`ForumsGetFeedOptions`** got its own dedicated `ForumFeedOption` model (new file)
  instead of reusing/loosening the shared `Forum` model — confirmed live that
  `/forums/feed-options` returns a consistently lighter shape (missing
  `forum_thread_count`/`forum_post_count`/`forum_prefixes`/`thread_default_prefix_id`/
  `thread_prefix_is_required`, plus one field `Forum` lacks: `has_children`) across all
  353 sampled entries, and `Forum` itself is still correctly required-heavy for
  `ForumsList`, so loosening it in place would have silently broken that endpoint's real
  guarantees instead.
- Everything else in the original "known-broken" list (`ForumsGrouped`, `FormsList`,
  `ChatboxGetLeaderboard`, `PaymentsPayoutServices`, `ProfileClaims`, `ProfileGet`,
  `ListStates`, `ListViewed`) — straightforward missing/mistyped-field fixes, same
  pattern as above; `ProfileGet.convertedBalance` is `float` not `int`
  (`int_from_float`, same class as `priceWithSellerFee`).

Every touched model file carries a dated "Nth pass" hand-patch note explaining what live
data showed and why — read the file's own docstring before touching a field it covers.
Live listings drift between runs (different sample of active items each time the suite
runs) — expect the occasional new one-off field to surface on a fresh run even after this
pass; the fix pattern is always the same (capture the real value via
`exc.errors(include_input=True)`, don't guess the type).

## Same bug class found again, different consumer (2026-07-05)

`lzt-open-source` (package `lzt_eventus`) has its own e2e suite
(`tests/e2e/test_real_lolzteam_api_e2e.py`, same `LZT_E2E_TOKEN`/`e2e`-marker convention)
that exercises `pylzt.Client` end to end. `test_get_lots_batch` failed with
`PydanticUserError: GetLotsBatch is not fully defined` — `methods/catalog.py` imported
`Sequence` only under `TYPE_CHECKING` but used it as a real field annotation on
`GetLotsBatch`, the exact same mistake `models/lot.py`'s `OrderBy` had (see above). Moved
`from collections.abc import Sequence` to a real top-level import. 21/21 passing now in
that repo's `tests/e2e/`. Worth a quick grep across the rest of `methods/*.py` for the
same `TYPE_CHECKING`-only-import-used-as-a-pydantic-field pattern — this is the second
occurrence, not necessarily the last.

## Generic `/batch` support added (2026-07-06)

`market_batch_requests.Batch` and `forum_batch_requests.BatchExecute` were codegen'd with
**zero request-body fields** — the spec's POST /batch body is a raw array of job objects,
a shape codegen's flat-field method generator can't capture, so both stubs were silently
unusable (calling them sent an empty array). Their guessed response model
(`Jobs`/`JobsJobId`) was also wrong, same class of bug as `Forum` reuse for
`ForumsGetFeedOptions`: it modeled `jobs` as one fixed `job_id` field when the real wire
shape is a dynamic map keyed by job id.

Added `BatchJob`/`BatchJobResult` + `build_generic_batch_request`/`parse_generic_batch_body`
in `lib/batch.py` (the lot-specific `_build_batch_request`/`_parse_batch_body` pair that
already existed are now thin wrappers over these, one implementation of the wire
mechanics). Hand-patched `Batch`/`BatchExecute` to take a real `jobs: Sequence[BatchJob]`
field (defaulted to `()` only so the still-generated, still-unwired
`GeneratedMarketFacade.batch()` / `GeneratedForumFacade.batch_execute()` facade stubs stay
type-valid — real calls always pass `jobs` explicitly; those two facade call sites needed a
documented `# type: ignore[arg-type]` since their declared `-> Jobs` return type no longer
matches, but the facade files themselves were left otherwise untouched — hand-patching a
whole ~1500-line generated file to fix two lines would block codegen from ever updating
the other ~100 methods in it). Added `Client.execute_batch(methods: Sequence[BaseMethod[T]])
-> list[T]`: **heterogeneous** methods batch together fine (verified live: `ListCategories`
+ two `GetLot`s in one POST /batch, each parsed by its own `parse_response`) — the earlier
"is it 1 type or many" question is answered live, not guessed.

Two more live-only findings while wiring this up, both would have silently produced wrong
results without a live check:

- **Job id `"0"` is treated as empty server-side** (a PHP/XenForo `empty("0") == true`
  quirk, near-certainly) — the response falls back to keying that job's result by its URI
  instead of the id sent. `execute_batch` uses 1-indexed job ids (`"1"`, `"2"`, ...), never
  `"0"`, to sidestep it. The pre-existing lot-batch path never hit this (real item ids are
  never literally `0`).
- **A batch hit's entry is NOT `{"_job_result": "ok", "item": {...}}` uniformly** — it's
  `{"_job_result": "ok", **<the batched endpoint's own top-level response body>}`. `GetLot`
  jobs happen to look `"item"`-wrapped only because `/{item_id}`'s own single-call response
  body legitimately has `"item"` as its top-level key; a batched `ListCategories` job comes
  back as `{"_job_result": "ok", "categories": [...]}` with no `"item"` key at all.
  `BatchJobResult.item` now strips only the `_job_result`/`_job_error` markers and hands the
  rest to the method's own `parse_response` unmodified — this is what makes heterogeneous
  batching work without per-endpoint special-casing.

## Separate, pre-existing bug found (not part of this pass) — fixed 2026-07-07

`tests/pylzt/test_methods_declarative.py` (6 tests) failed against the current
`BaseMethod` — they asserted dataclass-style construction (`GetLot(ItemId(7))` positional,
`dataclasses.is_dataclass(method)`) that hasn't been true since commit `7016a77` ("convert
the method layer from frozen dataclass to Pydantic"). The codegen gate
(`pipeline.validate()`) only runs `ruff` + `mypy` + an import smoke test — **it never runs
`pytest`**, so this regression shipped silently.

**Fixed 2026-07-07**: the tests now construct via keyword args and assert on
`pydantic.ValidationError` for the frozen check, matching the post-`7016a77` contract.
`.github/workflows/ci.yml` now runs `ruff check` + `ruff format --check` + `mypy` +
`pytest` on every push/PR, so this class of break fails CI instead of shipping silently.
The codegen gate itself is unchanged (still ruff+mypy+import only) — CI is the backstop
for the test suite, not the gate.

## Non-JSON 200 response silently lost, fixed 2026-07-09

Three endpoints declare their 200 response as `text/html`/`text/plain` with schema
`{"type": "string"}`, not `application/json`: `ListDownload` (`/user/{type}/download`),
`ManagingSteamPreview` (`/{item_id}/steam-preview`), `PublicCountLinesPlain`
(`/countLinesPlain`). Two independent bugs compounded:

- `dev/codegen/generator.py`'s `_resolve_response_schema` only looked at
  `content["application/json"]`, so all three resolved to no schema at all and
  generated as `BaseMethod[Any]` — technically not wrong, but hiding a real bug.
- `transport/session.py`'s `_decode` always called `response.json()` and returned
  `{}` on failure with no fallback — so even before the typing was fixed, calling any
  of these three endpoints silently discarded the real body and returned an empty dict.

**Fixed 2026-07-09**: `_resolve_response_schema` now falls back to whatever content
type IS declared when `application/json` is absent, so these three generate as
`BaseMethod[str]` (verified correct — regen with `_is_object` false on a bare
`{"type": "string"}` schema routes through the existing scalar/`_schema_type` path,
no new codegen branch needed). `Response` gained a `text: str | None` field, set by
`_decode` whenever the wire body isn't a JSON object; `BaseMethod.parse_response`'s
`passthrough` branch prefers `text` over the (now-meaningless) `{}` `body` when it's
set. This is a generic mechanism fix, not a per-endpoint hand-patch — any endpoint a
future spec update adds with the same non-JSON-string shape gets it for free.

## Mechanism reference

- **Method-level bug** (wrong `__url__`, wrong `__api__`, wrong `__rate_class__`) → fix
  `dev/codegen/generator.py` (it's a real generator bug, not spec drift) and rebuild.
- **Model-field bug** (spec declares a type the live API doesn't honor) → hand-patch the
  installed `src/pylzt/models/**.py` file, strip its auto-gen header, and record the
  fix here. The next `install()` for that API will skip (`_guard_no_clobber`) any staged
  file colliding with the now-hand-written one — warned and excluded, rest of the batch
  still installs — that's the intended guard, not a bug; resolve by reconciling this doc
  + the model before ever force-overwriting.
