# Утилиты фреймворка — расширенное меню идей (раунд 2)

Раунд 1 (топ-10, ранжированный) — в `00-overview.md`: watcher/diff-poller, тест-кит
(fake-transport), пресеты наблюдаемости, CLI, bulk-экспортёр, health-preflight,
VCR record/replay, query-builder над `LotFilter`, autobuy-sniper ⚠️ (отложено,
load-bearing), webhook-receiver ⚠️ (отложено, load-bearing). Параллельный
реестр middleware и детектор дрейфа схемы рассматривались и были отклонены
(реестр middleware: `MiddlewareManager` уже это покрывает; детектор дрейфа:
свёрнут в раунд 2 ниже с более чётким дизайном).

Этот файл — раунд 2: реально новые оси, а не вариации раунда 1.

Легенда: **S** ≤150 LOC · **M** 150–500 · **L** 500+. 🔴 = load-bearing
(деньги/auth/корректность rate-limit — полный architect-пол, не ponytail).

## Ось A — rate-limit / бюджет

1. **Синхронизация rate-limit с сервером (ground truth) — M 🔴** *(уже в очереди в `.plans/QUEUE.md`)*
   Кормить `RoundRobinTokenPool` реальным `system_info.rate_limit`
   (`limit`/`remaining`/`reset`/`bucket`), присутствующим в каждом теле ответа, вместо
   клиентской оценки по `ClientConfig.general_per_min`. Строится на `models/base.py`
   (парсит `system_info`), `token_pool/round_robin.py` (`RateBucketSet`),
   `transport/rate_limited.py` (фаза report уже есть). Самый высокий payoff —
   фундамент под #2–#3.
2. **Адаптивный регулятор параллелизма — M**
   Авто-тюнинг параллелизма в стиле AIMD по сигналу `remaining`/`reset` вместо
   статичного конфиг-параметра. Зависит от #1.
3. **Бенчмарк/soak-харнесс для тюнинга rate-limit — M**
   Синтетическая нагрузка через фейковый транспорт, строит кривую throughput
   к проценту 429, печатает рекомендуемые лимиты в минуту. Валидирует #2 офлайн.

## Ось B — persistence / координация

4. **Персистентное состояние token-pool — S/M**
   Карантин/бюджет/sticky-биндинги прокси переживают рестарт через существующий
   ABC `BaseStorage` (паттерн `BatchJobRecord` из `lib/batch.py`, переиспользован,
   не форкнут). Убирает стоимость холодного старта (401-разведка по всем токенам).
5. **Распределённый пул (бакеты на Redis) — L 🔴**
   Несколько процессов делят один rate-бюджет + состояние карантина через общий
   backend. Новый сиблинг рядом с `RoundRobinTokenPool`/`_StaticBearerPool` за
   ABC `BaseTokenPool`. 🔴 — атомарность бакета между процессами — реальный
   денежный/бан-риск.
6. **Примитив idempotency-store — S 🔴**
   `BaseIdempotencyStore` (memory/redis) для дедупа мутаций/вебхуков по
   upstream event id. Cross-cutting примитив по правилам проекта — извлекать
   вне зависимости от числа вызывающих. Enabler для отложенных в раунде 1
   autobuy-sniper/webhook-receiver (без него они небезопасны).

## Ось C — дрейф спеки / корректность

7. **CI-чекер дрейфа спеки — M**
   CLI сравнивает свежескрапленный фрагмент readme.io с закоммиченным
   `dev/generated/openapi/lzt_*.json`, падает в CI при изменении формы поля —
   ловит эволюцию апстрим-API раньше, чем это сделает багрепорт. Отдельный
   инструмент от VCR-рекордера раунда 1 (тот — рантайм-реплей, этот —
   статичный diff спеки).
8. **Рантайм-валидатор strict-mode — S**
   Опциональный тумблер: логировать типизированное предупреждение
   `SchemaDrift`, когда ответ несёт поле, которое Pydantic тихо выбросил,
   вместо молчаливого проглатывания. Динамический сосед #7 (тот — статичный).

## Ось D — dev-эргономика / debug

9. **Debug-middleware для dev-режима (эхо HAR/cURL) — S**
   `BaseMiddleware`, печатающий каждый запрос как готовый к вставке `curl`
   и/или пишущий HAR-файл. Чистая композиция на существующем `MiddlewareManager` —
   не новый реестр. Замыкает петлю с HAR-driven кодогеном проекта.
10. **Транспорт с fault-injection (chaos-тестирование) — S**
    Обёртка над `BaseTransport`, инжектящая 429/500/таймаут/битый JSON по
    вероятности — для проверки собственной retry/breaker-логики потребителя.
    В паре с тест-китом раунда 1 (тот даёт заготовленные happy-ответы, этот —
    заготовленный хаос).
11. **Фабрики моделей/фикстур — S/M**
    Фабрики в стиле `polyfactory`, сгенерированные на каждую модель
    (`LotFactory.build()`) — кодоген уже владеет каждой моделью, это
    естественный побочный выход.

## Ось E — оркестрация поверхjcn существующих примитивов

12. **Контекст-менеджер авто-батчинга — M**
    `async with client.batching():` — конкурентные вызовы внутри блока
    авто-коалесируются в `/batch` (окно, чанкинг по `MAX_BATCH_JOBS`). Делает
    батчинг бесплатным вместо opt-in-на-каждый-вызов.
13. **Резюмируемый экспортёр с персистентным курсором — S**
    Долгий экспорт сохраняет курсор `Paginator` в `BaseStorage`, переживает
    падение, продолжает с места остановки. Усиливает идею bulk-экспортёра
    из раунда 1.
14. **Планировщик периодического опроса — M**
    Лёгкий cron-подобный раннер, крутящий N watcher-фильтров (идея раунда 1)
    на интервалах с backpressure + graceful shutdown — субстрат, нужный
    любой автоматизации «следи и реагируй».
15. **Инспектор DLQ / упавших батчей — S**
    CLI поверх `batch_job_history`/`iter_pending_batch_jobs` для
    просмотра/ретрая/дропа застрявших батч-джобов — операционное окно в
    уже существующий at-least-once consume-commit loop, у которого сегодня
    нет вообще никакого UI.

## Ось F — proxy / инфра

16. **Адаптеры proxy-source — M**
    Готовые реализации `BaseProxySource` (файл, ENV, HTTP provider API вроде
    webshare/proxy6) с авто-рефилом мёртвых прокси. Сегодня потребителю
    приходится писать это с нуля вручную.
17. **ASGI health/readiness-пробы — S**
    Мини-роутер `/healthz`/`/readyz`, отдающий агрегированное состояние
    здоровья token-pool + прокси — тонкая обёртка над идеей health-preflight
    из раунда 1, но для k8s/деплой-тулинга вместо разового CLI-чека.

## Ось G — read-only аналитика (money-adjacent, но только чтение)

18. **Агрегатор портфеля/баланса по мульти-аккаунтам — M 🔴(read)**
    Свернуть баланс/активные лоты/продажи по каждому токену пула в одну
    типизированную сводку через `/batch`. Money-adjacent, но read-only —
    нужна аккуратная агрегация `Decimal`, но не полная строгость idempotency.
19. **Хранилище снапшотов истории цен — M**
    Персистить diff-события watcher'а раунда 1 в time-series-хранилище —
    запросы трендов «дешевле/дороже за 7д». Фундамент под будущий price-alert.

## Ось H — расширяемость

20. **Автообнаружение плагинов через entry-points — S**
    Авто-подхват сторонних middleware/метрик через entry points
    `importlib.metadata` (`pylzt.plugins`) вместо ручной регистрации.
    НЕ второй реестр (это было отклонено в раунде 1) — механизм автозагрузки,
    питающий единственный существующий шов `MiddlewareManager`/`BaseMetrics`,
    в духе aiogram/pytest.

## Ранжирование (самое полезное первым, по обоим раундам)

1. Синхронизация rate-limit (#1 здесь / в очереди) — 🔴, но это фундамент истины
2. Debug-middleware для dev-режима (#9) — почти нулевая цена, высокий DX-выигрыш
3. Персистентное состояние token-pool (#4) — убирает налог холодного старта
4. Контекст-менеджер авто-батчинга (#12) — бесплатный перформанс для
   существующих паттернов вызовов
5. CI-чекер дрейфа спеки (#7) — ловит поломку апстрима раньше пользователей
6. (топ-3 раунда 1: watcher/diff-poller → тест-кит → пресеты наблюдаемости)

**Извлекать в `libs/` вне зависимости от числа вызывающих** (cross-cutting
примитивы по конвенции проекта): idempotency-store (#6), адаптеры
proxy-source (#16).

**🔴 load-bearing, полная строгость:** #1, #5, #6, #18 (read-only, легче).
Плюс отложенные в раунде 1 autobuy-sniper / webhook-receiver.

**Lightweight/ponytail-OK:** #9, #10, #11, #13, #15, #17, #20 — все сидят на
существующих ABC-швах через композицию, без хирургии ядра.

**Новые оси относительно раунда 1:** persistence/координация (#4–6, #13, #19),
детекция дрейфа (#7–8), fault-injection (#10), планирование (#14),
автообнаружение плагинов (#20), адаптивный rate-control (#2–3). Раунд 1 был
про фичи для конечного пользователя; раунд 2 — про инфра-надёжность и
dev-тулинг — ту часть, которая реально отличает «фреймворк» от обёртки.
