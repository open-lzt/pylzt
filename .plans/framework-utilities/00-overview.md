# Утилиты фреймворка — архитектурное меню

**Что это:** ранжированное меню конкретных batteries-included инструментов, которые толкнут
pylzt от «типизированной обёртки» к экосистеме (в духе aiogram / FastAPI / Scrapy / discord.py).
Каждая идея опирается на *уже существующий* примитив — здесь нет greenfield-работы; каждый пункт —
рычаг поверх уже готовой машинерии (token pool, proxy pool, retry/breaker, TTL-кэш, коалесинг
`/batch`, `Paginator`, `BoundModel`, реестр типизированных ошибок, ABC `BaseMetrics`/transport,
`MiddlewareManager`, кодоген). Без DAG задач — это поверхность для принятия решения «какие 2-3 дальше».

Легенда: **S** ≤150 LOC · **M** 150-500 LOC · **L** 500+ LOC (отдельный пакет `libs/` или worktree).

---

## Тир 1 — флагманы (выбирать в первую очередь отсюда)

### 1. Watcher листингов / поток событий — **M** ⭐
Опрос фильтра по интервалу, диффинг со снапшотом, `yield` типизированных событий
`ListingAppeared` / `ListingPriceChanged` / `ListingGone` через `async for`.
**Почему подходит:** садится прямо на `Paginator` + `LotFilter` + read-поверхность + TTL-кэш
(дедуп-стор). Превращает read-only SDK в *реактивный* — самый крупный рычаг «фреймворк, не обёртка»,
и субстрат под все автоматизации ниже. Эмитить события, не вызывать транспорты напрямую (потребитель
сам подключает свой sink) — соответствует правилу проекта «события, а не транспорты».

### 2. Тест-кит для потребителей (`pylzt.testing`) — **M** ⭐
`FakeTransport` (реализует существующий ABC транспорта, отдаёт заготовленный JSON) + pytest-фикстуры
(`pylzt_client`, `respond_with(model)`, `raise_upstream(ErrorCode)`), поставляемые как extra.
**Почему подходит:** транспорт уже инжектируемый ABC, а кодоген уже владеет каждой response-моделью —
фейк, отдающий настоящие типизированные модели, почти бесплатен. Это *тот самый* сигнал
batteries-included: downstream-боты смогут юнит-тестироваться против pylzt без живого токена.
В паре с #7.

### 3. Пресеты наблюдаемости (`pylzt.observe`) — **S** ⭐
Три готовые реализации существующего DI-шва `BaseMetrics`: `OTelMetrics` (спаны на запрос/retry/
срабатывание breaker'а), `PrometheusMetrics` (счётчики/гистограммы), `StructlogMetrics` (биндит
`request_id`/`token_id`/`rate_class`). Однострочник: `Client(metrics=OTelMetrics())`.
**Почему подходит:** хук метрик уже существует и по умолчанию не используется — это чистый plug-in,
ноль изменений ядра, мгновенно делает SDK читаемым в проде. Самый дешёвый high-value выигрыш.

## Тир 2 — сильные, явные кандидаты

### 4. CLI `lzt` (`pylzt.cli`, Typer) — **M**
Превратить демо `__main__` в настоящий CLI: `lzt lot <id>`, `lzt search --category steam ...`,
`lzt tokens check`, `lzt export ...`, вывод `--json`.
**Почему подходит:** `__main__.py` + sync-фасад (в разработке) уже есть; CLI — тонкая
argparse/Typer-обёртка над `SyncClient`. Обязательный атрибут «ощущения экосистемы».

### 5. Bulk-экспортёр с учётом rate-limit — **M**
`export(filter, sink=CSV/JSONL/Parquet)`, который вычерпывает `Paginator`, уважая бакеты по
каждому `RateClass` и коалесируя детальные запросы через `/batch`.
**Почему подходит:** складывает бакеты token-pool + `Paginator` + generic `/batch` — все три уже
построены; это оркестрация, а не новая машинерия. Самая «очевидная польза» на LOC.

### 6. Preflight-проверка здоровья токенов/прокси — **S**
`await client.health()` → валидирует каждый токен (живой пинг в духе `/checkAccess`), репортит
мёртвые/лимитированные/с низким балансом токены и состояние breaker'а по каждому прокси.
**Почему подходит:** читает прямо из token pool + состояния breaker'а proxy-pool, которые уже
трекают это внутри — просто выводит наружу. Небольшая, и первое, что нужно любой мульти-аккаунт-операции.

### 7. VCR-style запись и воспроизведение — **M**
Транспорт-декоратор, записывающий живые ответы в кассету, и транспорт-плеер, отдающий их обратно —
для детерминированных тестов и офлайн-демо.
**Почему подходит:** тот же шов ABC транспорта, что и #2 (обёртка-композиция, не форк); скрейпер
кодогена уже доказывает паттерн «захватить реальные ответы». Естественный компаньон тест-кита.

### 8. Декларативный query-builder — **M**
Флюентный, валидируемый по схеме слой над `LotFilter`:
`Q.steam().price(lt=50).origin(BRUTE).order(NEWEST)`, валидируется против живой `FilterSchema`
из `category_params`.
**Почему подходит:** `LotFilter` + `FilterSchema` на категорию уже существуют — билдер делает
*динамические* пер-категорийные параметры обнаружимыми/типизированными вместо сырых dict.
Множитель эргономики для #1/#5.

## Тир 3 — мощные, но тяжелее / уже

### 9. Autobuy / sniper движок правил — **L** ⚠️ load-bearing
Декларативные правила (`когда price < X и origin в {...} → купить`), питаемые watcher'ом #1,
исполняющиеся через методы покупки с идемпотентностью и dry-run режимом.
**Почему подходит:** композиция watcher (#1) + `execute` + типизированные ошибки. **Но** это
касается денег/заказов — полный architect-пол, не ponytail: idempotency-ключ на срабатывание
правила, оптимистичный дедуп, `--dry-run` по умолчанию. Сначала #1; это его killer-app,
поставляется осознанно.

### 10. Хелпер приёма вебхуков — **M**
ASGI мини-приложение (или хендлер), верифицирующее + парсящее **Invoice Webhook** маркетплейса
в типизированное событие `InvoicePaid`, дедупленное по upstream event id.
**Почему подходит:** расширяет философию типизированных моделей и ошибок на входящее направление.
**Сначала проверить:** приватный модуль `repo-webhooks`, вероятно, уже владеет доставкой/дедупом/HMAC —
переиспользовать его, не изобретать заново. Payment-adjacent → верификация подписи обязательна,
не опциональна.

## Отклонено / свёрнуто

- **Система регистрации плагинов/middleware** — *отклонено.* `MiddlewareManager` уже существует;
  второй слой регистрации — это форк (правило проекта §3, единый источник истины). Если эргономики
  middleware не хватает — расширять существующий менеджер, не добавлять параллельный.
- **Отдельный детектор дрейфа схемы** — *свёрнуто в #7.* Сравнение живых ответов со сгенерированными
  моделями — естественный побочный продукт записывающей ветки + кодогена; пока не отдельный
  инструмент (один вызывающий).

## Рекомендация — поставить эти 3 первыми

1. **#3 Пресеты наблюдаемости (S)** — самое дешёвое, ноль риска для ядра, задействует
   неиспользуемый шов `BaseMetrics`; делает всё последующее отлаживаемым в проде. Сделать первым
   как включающий слой.
2. **#1 Watcher листингов (M)** — флагман, переопределяющий pylzt из read-SDK в реактивный
   фреймворк, субстрат для #5/#8/#9. Наибольший архитектурный выигрыш.
3. **#2 Тест-кит (M)** — самый явный сигнал batteries-included для *усыновителей*; низкий риск
   (ABC транспорта уже есть), и именно это позволяет комьюнити уверенно строить поверх pylzt.

**Почему именно эта тройка:** один дешёвый enabler (observability), один флагман, меняющий
категорию SDK (watcher), один мультипликатор адопции (тест-кит) — вместе они двигают позиционирование
«фреймворк, не обёртка», которое уже заявляет переписанный README, при этом все три едут на
существующих швах (хук метрик, Paginator+кэш, ABC транспорта) без хирургии ядра. #9/#10 отложить
до появления бюджета на ревью денежных путей — обе сильные, но load-bearing.
